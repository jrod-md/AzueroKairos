1. Resumen ejecutivo

AgroShield evoluciono desde una idea inicial de monitoreo de calidad hidrica hacia una propuesta mas defendible: un Satellite Confidence Semaforo y un AgroShield Confidence Brief para apoyar decisiones agricolas en Azuero usando evidencia Copernicus.

La idea central ya no es afirmar contaminacion, calidad completa del agua o deteccion quimica. La idea central es determinar si una observacion Sentinel-2 tiene suficiente confianza para apoyar una lectura exploratoria hidro-sedimentaria o si la decision responsable es no inferir y recomendar verificacion local.

La posicion tecnica actual es:

GO TECHNICAL / ADJUST SCIENTIFIC

Esto significa que la base tecnica funciona, pero la historia cientifica debe mantenerse cautelosa. Hay suficiente evidencia para continuar AgroShield, pero no suficiente para afirmar validacion de una crisis de junio 2025 ni deteccion de contaminantes.


2. Que se probo tecnicamente en el spike descartable

El spike interno, privado, descartable y no entregable demostro lo siguiente:

- CDSE authentication funciona.
- Copernicus Data Space Ecosystem / Sentinel Hub Statistical API funciona.
- Catalog API funciona para encontrar adquisiciones Sentinel-2 L2A.
- La reproyeccion CRS84/lonlat a EPSG:32617 funciona.
- Statistical API devuelve estadisticas MNDWI y NDTI.
- El pipeline puede guardar respuestas crudas JSON.
- El pipeline puede producir CSV procesado.
- Se puede calcular sampleCount, noDataCount y validPercent.
- Se puede clasificar la confianza de cada observacion como usable, low_confidence o invalid.
- El sistema puede operar con cache-first para reducir riesgo de rate limits.

Esto prueba viabilidad tecnica de acceso, procesamiento y resumen de evidencia Sentinel-2. No prueba una validacion operacional.


3. Que NO se probo

No se probo:

- Deteccion de atrazina.
- Deteccion de pesticidas.
- Deteccion de contaminacion quimica.
- Deteccion de patogenos.
- Deteccion de metales pesados.
- Calidad hidrica completa.
- Agua segura.
- Validacion fuerte de la crisis de junio 2025.
- Sistema operacional listo.
- Decision automatica de cierre de toma.
- Suspension obligatoria de riego.
- Prediccion garantizada.
- Reemplazo de laboratorio o muestreo de campo.

La lectura correcta es:

"Evidencia satelital exploratoria asociada a riesgo hidro-sedimentario observable."


4. Resultados tecnicos principales

Los resultados del technical ceiling test indican:

AOI broad:
- 10 filas totales.
- 4 filas usable.
- 0 filas low_confidence.
- 6 filas invalid.
- 2 fechas usable.
- Mean validPercent aproximado: 31.05.
- Lectura: tecnicamente devuelve mas datos, pero probablemente diluye la senal del corredor ribereno por mezclar zonas urbanas, agricolas y riberenas.

AOI corridor_wide:
- 10 filas totales.
- 4 filas usable.
- 0 filas low_confidence.
- 6 filas invalid.
- 2 fechas usable.
- Mean validPercent aproximado: 18.89.
- Lectura: mejor compromiso cientifico actual entre cobertura suficiente y defensibilidad agricola-riberena.

AOI river:
- 10 filas totales.
- 2 filas usable.
- 2 filas low_confidence.
- 6 filas invalid.
- 1 fecha usable.
- Mean validPercent aproximado: 14.92.
- Lectura: util como zoom tecnico, pero demasiado fragil como AOI primario.

Conclusion sobre AOI:

- broad sirve como referencia tecnica, pero no como narrativa principal.
- corridor_wide debe ser el AOI primario del producto.
- river debe ser secundario o zoom de explicacion.


5. Producto defendible actual

El producto mas defendible no es un dashboard generico ni un detector de contaminacion.

El producto defendible es:

AgroShield Confidence Brief

Una herramienta que, para una fecha y AOI, muestra:

- AOI seleccionado.
- Fecha Sentinel-2 confirmada.
- Porcentaje de pixeles validos.
- MNDWI.
- NDTI.
- Confidence state: usable, low_confidence o invalid.
- Risk state exploratorio: normal, watch, review o do_not_infer.
- Explicacion de por que el sistema llego a ese estado.
- Recomendacion agricola proporcional.
- Limitaciones visibles.
- Modo de fuente: cache o live.
- Registro trazable tipo prediction ledger.

La propuesta central:

"AgroShield no solo muestra mapas. Muestra si la evidencia satelital es suficientemente confiable para apoyar una decision agricola."


6. Por que esto puede ser fuerte para el hackathon

El hackathon esta orientado a agricultura sostenible y seguridad alimentaria con observacion de la Tierra. AgroShield encaja si se presenta desde el problema alimentario:

- El agua del Rio La Villa funciona como infraestructura agricola.
- El agua afecta riego, lavado, inocuidad, agroexportacion y continuidad productiva.
- Las decisiones agricolas pueden ser peligrosas si se basan en evidencia satelital mala tratada como si fuera buena.
- La incertidumbre tambien debe convertirse en decision: no inferir, revisar o verificar localmente.

El diferencial competitivo no debe ser una visualizacion bonita. Debe ser la disciplina del producto:

- Primero confianza.
- Luego riesgo exploratorio.
- Luego recomendacion proporcional.
- Siempre limitaciones.
- Siempre claim firewall.


7. Por que todavia no es automaticamente ganador

La base tecnica es buena, pero aun no basta para asegurar que el proyecto gane.

Razon principal:

La senal cientifica todavia no es suficientemente fuerte para decir que se identifico una anomalia hidro-sedimentaria clara alrededor del 11 de junio de 2025.

Riesgos actuales:

- Hay pocas fechas usable en la muestra actual.
- Muchas fechas son invalid por nubes/noData.
- broad da mas datos pero diluye la senal.
- river es cientificamente cercano al cauce, pero tiene menos pixeles utiles.
- corridor_wide parece el mejor compromiso, pero necesita mas fechas y comparaciones.
- NDTI puede no diferenciar lo suficiente.
- Si el producto se presenta como deteccion de contaminacion, seria vulnerable.

Conclusion honesta:

AgroShield puede ser muy competitivo si se presenta como sistema de confianza satelital y decision agricola bajo incertidumbre. No debe presentarse como detector de crisis ni detector quimico.


8. Claims permitidos

Se puede decir:

- Evidencia satelital derivada de Copernicus.
- Semaforo de confianza satelital.
- Screening exploratorio de senal hidro-sedimentaria.
- Recomendacion de revision o verificacion local.
- No inferir cuando la evidencia satelital es insuficiente.
- Herramienta de apoyo a decisiones agricolas.
- MVP reproducible con cache de resultados Copernicus.
- No reemplaza muestreo de campo ni laboratorio.
- Disenado para reducir decisiones ciegas en riego e inocuidad agricola.

10. MVP recomendado

El MVP oficial debe construirse desde cero durante la ventana oficial de competencia.

MVP core recomendado:

Fase 0: Evidence Kernel
- Cargar datos Copernicus cacheados/generados oficialmente.
- Normalizar AOI, fecha, indices y metricas.

Fase 1: Satellite Confidence Semaforo
- Clasificar observaciones como usable, low_confidence o invalid.

Fase 2: Hydro-Sedimentary Screening
- Usar MNDWI y NDTI como indicadores exploratorios, solo si la confianza lo permite.

Fase 3: Agricultural Decision Brief
- Generar una recomendacion agricola proporcional.

Fase 4: Risk Explainability Panel
- Explicar reglas, limites y razon de la decision.

Fases fuertes si hay tiempo:

- Multi-AOI Decision Mode.
- Evidence Packet Export.
- AI BriefWriter.
- Claim Firewall.
- Prediction Ledger.
- Regional Replication Pack.


11. AI en AgroShield

La IA puede hacer el producto mas claro, pero no debe ser el motor cientifico.

Uso permitido de IA:

- Redactar briefs por audiencia.
- Traducir resultados estructurados a lenguaje de productor, tecnico, jurado o institucion.
- Resumir limitaciones.
- Mejorar claridad de recomendaciones.

Uso prohibido de IA:

- Calcular riesgo.
- Cambiar risk_state.
- Cambiar confidence_state.
- Inventar fuentes.
- Afirmar contaminacion.
- Emitir ordenes obligatorias.
- Saltarse el claim firewall.

Si la IA falla, el producto debe seguir funcionando con plantillas locales.


12. Claim Firewall

El Claim Firewall es un diferenciador importante.

Debe bloquear textos que sugieran:

- Deteccion quimica.
- Agua segura.
- Validacion de crisis.
- Tiempo real.
- Cierre automatico.
- Suspension obligatoria.
- Prediccion garantizada.
- Sustitucion de laboratorio.

Si detecta un claim peligroso, debe reemplazar el texto por una version segura:

"Esta salida resume evidencia satelital Copernicus disponible. No detecta quimicos ni reemplaza verificacion local o laboratorio."


13. Otras ideas evaluadas

Opcion 1: AgroShield Confidence Brief
- Mejor opcion actual.
- Usa lo que ya se probo.
- Diferencia clara frente a dashboards.
- Mantiene claims defendibles.

Opcion 2: AgroShield for Agroexport Compliance
- Variante B2B.
- Pre-screening para priorizar revision de agua usada en lavado/riego.
- Puede ser comercialmente fuerte.

Opcion 3: Azuero Food Water Sentinel
- Variante B2G.
- Enfocada en municipios, instituciones o autoridades.
- Buena para seguridad alimentaria territorial.

Opcion 4: Copernicus Crop-Water Confidence Network
- Version regional.
- Mas ambiciosa.
- Riesgo de sonar demasiado grande si no se controla.

Opcion 5: Pivot a sequia/estres hidrico agricola
- Posiblemente mas facil tecnicamente con NDVI/NDMI y clima.
- Menos original.
- Riesgo de convertirse en dashboard generico.

Decision recomendada:

No pivotear todavia. Hacer AgroShield mas afilado, no mas amplio.


14. Pruebas adicionales recomendadas

Las pruebas adicionales deben ser pocas y quirurgicas.

Prueba 1: Mas fechas corridor_wide
- Correr mas fechas catalog-confirmed entre mayo y julio 2025.
- Priorizar fechas cercanas a junio.
- Usar cache-first.
- Dormir entre requests.
- Evitar rate limits.

Prueba 2: Baseline vs candidate
- Seleccionar fechas usable antes y despues.
- Comparar MNDWI/NDTI.
- No forzar anomalia si no aparece.

Prueba 3: corridor_wide vs river
- Usar river como zoom secundario.
- Pregunta: el zoom confirma, contradice o queda invalid?

Prueba 4: Sensibilidad del AOI
- Mover ligeramente corridor_wide.
- Ver si la senal cambia demasiado.
- Si cambia mucho, documentar incertidumbre.

Prueba 5: NDTI como indicador secundario
- Evaluar si NDTI realmente aporta contraste.
- Si no aporta, mantenerlo como proxy exploratorio secundario.
- No basar el producto completo en NDTI.


15. Criterio para decidir despues de mas pruebas

Si corridor_wide obtiene varias fechas usable y algun contraste interpretable:

Continuar con AgroShield como Satellite Confidence Semaforo + hydro-sedimentary exploratory screening.

Si corridor_wide obtiene fechas usable pero NDTI no diferencia:

Continuar con AgroShield como Satellite Confidence Semaforo + MNDWI/wetness confidence + NDTI secundario.

Si casi todo sale invalid:

Replantear AOI o ampliar corredor. No pivotear de inmediato.

Si todos los AOI fallan despues de pruebas razonables:

Considerar pivot tecnico hacia sequia/estres hidrico agricola o otro producto Copernicus de seguridad alimentaria.


16. Veredicto actual

AgroShield no esta listo para decir "validamos una crisis".

AgroShield si esta listo para convertirse en un producto serio si se construye alrededor de esta tesis:

"En agricultura, usar mala evidencia satelital como si fuera buena tambien es un riesgo. AgroShield convierte Copernicus en un semaforo de confianza y un brief agricola que indica cuando interpretar, cuando revisar y cuando no inferir."

Eso es tecnicamente honesto, cientificamente defendible y potencialmente competitivo.


17. Proximo paso recomendado

Antes de escribir mas documentos o construir el MVP, conviene correr una ronda corta de pruebas:

- Solo corridor_wide como AOI primario.
- Fechas catalog-confirmed entre mayo y julio 2025.
- Maximo pequeno de requests.
- Cache-first.
- Sleep alto.
- Comparacion baseline/candidate.
- Sin claims de validacion.

El objetivo no es probar una narrativa a la fuerza. El objetivo es medir si hay suficiente senal para fortalecer el demo o si el producto debe centrarse casi por completo en confianza satelital.
